Чтобы агент в Cursor мог САМ (терминал + скрипты), без входа в твои аккаунты Vercel/Supabase в браузере
================================================================================

1) Файл .env в корне проекта (как в .env.example):
   - DATABASE_URL — как сейчас (можно pooler / prisma+ если так у тебя заведено для рантайма).
   - DIRECT_URL — ОБЫЧНЫЙ postgresql://... на Postgres (Supabase «Direct», порт 5432), БЕЗ prisma+.
     Без этого агент не сможет применить миграции и в таблице чата будет ошибка «таблица не существует».

2) Один раз на этой машине, из папки проекта:
   npx prisma migrate deploy
   (если нужны тестовые логины из сида: npx tsx prisma/seed.ts)

3) Дальше агент сам может:
   npm run chat:test-send -- Проверка от агента
   npm run lint
   npm run build
   npm run test:e2e   (нужен запущенный npm run dev; SUPER_ADMIN по умолчанию Den / admin123 — см. clean-data.md)

4) Чтобы агент «как пользователь» кликал по сайту:
   в настройках Cursor подключи Playwright MCP или Chrome DevTools MCP,
   держи npm run dev — тогда можно открывать localhost и проверять тосты/чат визуально.

5) Чего НЕ требовать от агента:
   - не передавать пароли от прод-Supabase/Vercel в чат;
   - для прод-операций — только твои ручные действия или отдельный слабый тестовый проект БД.

6) E2e на проде / Vercel: PLAYWRIGHT_SUPERADMIN_USER=Den, PLAYWRIGHT_SUPERADMIN_PASSWORD=admin123.

7) Аватары на Vercel (2026-05):
   - В панели Vercel → Storage: store thaiinvest-v3-avatars, переменная BLOB_READ_WRITE_TOKEN в Production И Preview.
   - После деплоя: npm run smoke:prod:avatar (по умолчанию Den/admin123 или PLAYWRIGHT_SUPERADMIN_*).
   - Если супер-админ на проде не входит: npm run db:reset-admin-password:prod (проверка), затем с CONFIRM=yes — :apply (см. .env.example).
   - Старые пути /uploads/ в БД: npm run db:list-stale-avatars:prod
