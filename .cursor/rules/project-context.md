## Правило №0 — ритуал перед любой работой (обязательно)

**В начале сессии и перед любыми правками в репозитории** — не заменять этот шаг «памятью» из чата:

1. **Перечитать все файлы из `.cursor/rules/`** — в том числе этот `project-context.md`, все релевантные `.mdc` и `.md` (например `clean-data.md`, `investor-dashboard-protected.md`).
2. **Перечитать `SESSION_LOG.md`** в корне репозитория.
3. **Перечитать `Для анализа/ГЛОССАРИЙ.txt`** (короткие определения ролей и терминов).
4. **Зафиксировать контур по умолчанию** (и не противоречить ему без явного указания пользователя):
   - **База данных — локальная** для разработки; **Docker для БД не обязателен** (Postgres может быть без контейнера).
   - **Supabase** — контур **продакшена** (удалённый хостинг); для локальной разработки **не** считать его источником данных, пока пользователь явно не скажет обратное.
   - **Валюта в продукте — тайские баты (฿)**.
   - **Переменные окружения:** цепочка как у Next и `scripts/ensure-local-db.mjs` — сначала **`.env`**, затем **`.env.local`** с **override**. Скрипты, Prisma CLI и разовые `tsx`-проверки **не** запускать только с `.env`, если в проекте используется `.env.local` (иначе возможна «пустая» или другая БД).

Дальше по чеклисту — §0.1 и при работе с БД — §6.2.

## 7) Учёт **`accrued`**, **`paid`** и недельный леджер (единственный источник истины)

**Запрещено** вводить второй способ расчёта «начислено» или недельных сумм в обход нижеуказанного кода (никаких параллельных формул `body * rate * weeks` и т.п.).

**Обязательная синхронизация:** при **любом** создании, удалении или редактировании данных, которые участвуют в леджере (**`Payment`**, **`BodyTopUpRequest`** — статус/сумма/даты, завершение выплаты и т.д.), после успешной мутации вызывать **`syncSingleInvestorAccruedAndPaidFromLedger`** (`lib/business-rate-accrual-recalc.ts`) для затронутого **`investorId`**, чтобы в БД сразу попали **`accrued`** = **`computeInvestorAccruedEndFromLedger`** и **`paid`** = **`computeInvestorPaidCompletedTotal`**. Не полагаться только на отложенный **`scheduleBusinessRateRecalc`** для согласованности после удаления строк.

### 7.1 Поле **`Investor.accrued`**

- Считается **только** через **`buildWeeklyLedgerRows`** → последняя строка **`accruedEnd`**, обёрнутая в **`computeInvestorAccruedEndFromLedger`** (`lib/investor-accrued-ledger.ts`).
- Учитываются **только закрытые недели**; текущая открытая неделя **не добавляет** процент в остаток; выплаты текущей недели **уменьшают** остаток по правилам леджера.
- Итог в БД — **целые баты** (`Math.round` / без дробной части в продуктовом смысле).
- Массовый пересчёт: **`recalculateInvestorAccruedFromRateHistory`** (`lib/business-rate-accrual-recalc.ts`) и очередь **`scheduleBusinessRateRecalc`** (изменения **`RateHistory`** и т.п.). Точечная синхронизация одной позиции — **`syncSingleInvestorAccruedAndPaidFromLedger`** (тот же канон, что и в **`PATCH /api/investors/[id]`** при пересчёте).

### 7.2 Поле **`Investor.paid`**

- Отражает **сумму всех завершённых выплат** (`Payment.status === "completed"`), **всех типов** — см. **`computeInvestorPaidCompletedTotal`** в **`lib/investor-accrued-ledger.ts`**.
- При **`shouldRecalculateAccrued`** в **`PATCH /api/investors/[id]`** и в **`recalculateInvestorAccruedFromRateHistory`** поле **`paid`** пересчитывается через **`computeInvestorPaidCompletedTotal`** (согласовано с **`Payment`**). Если в том же запросе SUPER_ADMIN явно передал **`paid`**, но **не** сработал автопересчёт **`accrued`** — сохраняется значение **`paid`** из запроса.

### 7.3 Пополнения тела (**`BodyTopUpRequest`**, принято инвестором)

- В леджере сумма вступает в силу с **`getNextMonday(startOfDay(effectiveAt))`**, где **`effectiveAt`** = `requestDate ?? decidedAt ?? createdAt` (см. **`ledgerAcceptedTopUpsFromPrismaRows`**).
- База для недель **до** этого понедельника — без этой суммы; **после** — с увеличенным телом (детали в комментарии к **`buildWeeklyLedgerRows`**, `lib/weekly-ledger-rows.ts`).

### 7.4 Лента «Финансы» — порядок строк (`sortAt`)

- **Начисление за неделю (`week_accrual`):** в ленте **`sortAt` = `weekEnd`** (конец закрытого интервала в данных леджера), а не `weekStart`, чтобы выплаты и пополнения **внутри** той же календарной недели с датой **после** понедельня не оказывались **выше** строки итога начисления за эту неделю при сортировке по убыванию. **Фильтр периода** на клиенте использует **`operationPeriodAnchorIso`** → для недель **`weekStart`**, чтобы строка не выпадала из выбранного календарного месяца.
- **Синтетическое «начальное тело при открытии»:** **`sortAt`** = **`getWeekStartMonday(startOfDay(min(entryDate, activationDate))) − 1 ms`** (`GET /api/investors/operations-history`).
- **Выплаты (`payment`):** **`sortAt`** = **`createdAt`** заявки (момент подачи), не дата завершения — см. комментарий в обработчике; иначе правки/завершение в другом месяце ломают визуальный порядок относительно недель.
