### 0.1 Режим работы агента (обязательно)

- **Перед любыми правками в репозитории** перечитать этот файл (`project-context.md`), применимые `.mdc` и применимые `.md` из `.cursor/rules/` (в т.ч. `clean-data.md`, `investor-dashboard-protected.md`).

- Исключение: если без одного уточнения реализация однозначно сломает учёт или безопасность — **один** конкретный вопрос, без простыни опций.

### 0.2 Эталон: правки выплат и дат (SUPER_ADMIN) — 2026-05-08

- **Откат по этапу выплаты / ошибка суммы:** возврат заявки на **шаг того, кому принадлежит решение на этом этапе** (например к Семёну как владельцу на шаге суммы/одобрения); он правит сумму; дальше снова обычная цепочка согласований. В комментарии фиксировать суть правки (например: исправлена сумма с 200 на 2000).
- **Правка даты / старых данных** (ошиблись в дате, подставляли исторические даты): согласование и исполнение — **по владельцу данных** (кто отвечает за этот факт); супер‑админ может вносить правку там, где это его зона ответственности, по вашей логике сценария.
- **Где в коде:** Prisma `PaymentCorrectionProposal`; `GET`/`POST` `/api/payment-correction-proposals`; `PATCH` `/api/payment-correction-proposals/[id]`; очередь `components/dashboard/finance/PaymentCorrectionQueue.tsx` (на странице «Финансы»); **`FinanceOperationDetailModal.tsx`** — у SUPER_ADMIN **инлайн** черновики дат и суммы с отправкой на согласование (галка), отдельный блок **«Удалить операцию»** с подтверждением через `AppDialogsProvider` (`DELETE` `/api/payments/[id]`); оповещение адресату — сообщение в чат (`lib/payment-correction-notify.ts`). После принятия заявки на правку: для завершённых процентов/тела выполняется **обратная проводка** по позиции и смена статуса/дат по полезной нагрузке запроса.

### 0.3 Полномочия SUPER_ADMIN и эксплуатация Prisma

- По договорённости с владельцем продукта учётка **SUPER_ADMIN** (платформа; после сида по умолчанию логин **`admin`**) может **всё операционное**: удалять операции из истории (где это допускается бизнес‑правилами UI/API), инициировать правки **сумм и дат** (инлайн и через очередь согласований). Это не отменяет ограничений на отдельные типы (например завершённое закрытие позиции — см. подсказку в модалке).
- **Пароли и секреты** в репозиторий не коммитить и не дублировать в этом файле; для проверок — переменные окружения (например `PLAYWRIGHT_SUPERADMIN_USER` / `PLAYWRIGHT_SUPERADMIN_PASSWORD`).
- После **`prisma migrate`** или **`prisma generate`** нужно **перезапустить dev‑сервер**, иначе singleton Prisma может отдавать **`paymentCorrectionProposal` = undefined** → пустые списки в GET или **503** на POST/PATCH правок; см. `lib/payment-correction-proposal-delegate.ts`.

## 2) OWNER — отдельный экран, не ломая инвестора

- Для OWNER (Sam) развиваем `OwnerPremiumDashboard` и всё, что относится к OWNER‑UX.
- Встраивание в `/dashboard` делается так, чтобы инвесторский экран не менялся.

### 2.1) OWNERSHIP — назначение `ownerId` при создании инвестора

- В **общей** сети (`isPrivate: false`) → **`ownerId` = первый активный OWNER** в БД (по возрастанию `id`; в типичном seed после сида это **Sam**), чтобы позиции общей сети, созданные SUPER_ADMIN, числились на операционного владельца и отображались у него на главной и в реестре.
- В **личной** сети SUPER_ADMIN (`isPrivate: true`) → **`ownerId` = SUPER_ADMIN**, выполнивший создание (в типичном seed — **`admin`**).
- Реализация: **`app/api/investors/route.ts`**, функция **`resolveOwnerIdForNewInvestor`** (вызов при создании инвестора в этом маршруте).
- **Запрещено** менять эту логику и смысл правила **без явного разрешения** пользователя.

- Не использовать seed/скрипты, которые создают фиксированных пользователей по `username` (иначе при переименовании “освободившееся” имя будет создано заново, появится “копия” с той же ролью).
- `scripts/create-admin.ts`: не дублировать учётку **`admin`** (SUPER_ADMIN); она создаётся через `prisma/seed.ts` при необходимости. Скрипт дополняет окружение (например OWNER **Sam**), не подменяет супер-админа.
- `prisma/seed.ts`: seed не upsert'ит по фиксированным `username`, а проверяет наличие хотя бы одного пользователя по ролям.

Правило: если снова появляются лаги 10–60с — сначала проверять, не ходит ли dev в удалённый `DATABASE_URL`; дальше — кеш/агрегации, а не “ускорять UI скелетонами”.

### 6.1 Опционально: локальная БД через Docker

- По умолчанию dev может ходить в **удалённую** БД из `.env` (например Supabase) — без ручного переключения.
- Если понадобится локальный Postgres: **Docker Desktop** + `docker-compose.dev-db.yml` (порт **54321**), шаблон строк — `.env.local.example` (переименовать/скопировать в `.env.local` только при использовании Docker).
- Команды: `npm run db:dev:install-docker` (UAC), затем `npm run db:dev:bootstrap`, миграции/seed как в `package.json`.
