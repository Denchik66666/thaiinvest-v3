4) ЧТО ТЕБЕ ДЕЛАТЬ ТЕПЕРЬ (ПРОСТОЙ РЕГЛАМЕНТ)

Правильный процесс правок:

1. Любая новая правка -> в staging.
2. Проверка (визуально + сценарии).
3. Pull Request из staging в main.
4. После merge в main обновление автоматически уходит в прод.

Перед релизом короткая проверка:

- вход/выход;
- Главная;
- Финансы;
- Отчеты;
- запрос на вывод и изменение статуса.

8) ЧАТ / SEGA (АНАЛИЗ И ПРАВКИ, 17.04.2026)

Проблема: у пользователя с ролью INVESTOR (в т.ч. Sega) на экране чата иногда показывалось
«Не найден владелец сети…» даже при исправной базе.

Причины (две):
  A) API раньше брал владельца только по Investor.investorUserId. Если в прод-базе не было
     строки Investor для этого User — defaultPeer был пустой.
     Исправление: lib/chat-network.ts + /api/chat/context, /api/chat/directory, /api/chat/messages —
     учитываются investorUserId И linkedUserId; если позиции нет — fallback на первого OWNER.
  B) На клиенте (app/dashboard/chat/page.tsx) ошибка показывалась пока React Query ещё не получил
     ответ /api/chat/context: resolvedPeer был null в момент загрузки и UI сразу рисовал ошибку.
     Исправление: пока ctxPending — текст «Загрузка чата…»; ошибка — только после успешной
     загрузки, если defaultPeer по-прежнему null (нет OWNER в БД).

Сид: prisma/seed.ts создаёт пользователя Sega и при необходимости Investor с investorUserId=Sega
и ownerId=semen (OWNER).

Важно: чтобы правки A и B попали на телефон, нужен деплой на Vercel (merge в main по вашему процессу).
