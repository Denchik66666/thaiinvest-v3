ОДНА база для локалки, Vercel и Prisma
=======================================

Частая причина «ничего не работает как в проде»: в файле .env указан ОДИН
проект Supabase, а на Vercel в переменных — ДРУГОЙ (другой ref в имени
хоста или в пользователе postgres.XXXX).

Как проверить за 30 секунд
--------------------------
1) В Vercel → твой проект → Settings → Environment Variables → скопируй
   только ФРАГМЕНТ хоста из DATABASE_URL (например ...pooler.supabase.com
   и кусок postgres.ДЛИННЫЙ_ID).

2) Открой локальный .env и сравни: должен быть ТОТ ЖЕ проект / тот же ref.

3) В терминале папки проекта выполни:
   npm run db:verify

   Должны появиться многие таблицы: User, Investor, Payment, …
   Если в списке только «User» — это ПУСТАЯ или ЧУЖАЯ база, не та где
   твои инвесторы и чат. Исправь DATABASE_URL и DIRECT_URL на проект
   из Vercel (две строки из Supabase Connect для ЭТОГО проекта).

DIRECT_URL для Prisma CLI
-------------------------
• На Windows часто не открывается прямой db....supabase.co:5432 (IPv6).
  Тогда DIRECT_URL = session pooler на порту 5432 (тот же pooler-хост,
  что в DATABASE_URL, пользователь postgres.REF, порт 5432).

Миграции Prisma
---------------
• migration_lock.toml в репозитории должен быть postgresql (не sqlite).
• Если база уже живая и история миграций не совпадает — не гоните
  слепо migrate deploy на прод без бэкапа; лучше один раз выровнять
  с DBA или через новый проект Supabase под чистые миграции.
