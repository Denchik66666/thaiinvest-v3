Анализ старого проекта Supabase — чтобы «ничего не рухнуло»
============================================================

Я (агент) не могу зайти в твой Supabase в браузере. Но можно БЕЗОПАСНО
проанализировать базу с твоего компьютера: только ЧТЕНИЕ, без удалений.

Что сделать по шагам
--------------------

1) В Supabase для СТАРОГО проекта сделай резервную копию на всякий случай:
   Dashboard → Database → Backups (или экспорт через SQL / план Supabase).

2) В Cursor в файле .env временно укажи DIRECT_URL (и при необходимости
   DATABASE_URL) именно на ЭТОТ старый проект (строки из Connect).

3) В терминале папки проекта выполни:
   npm run db:analyze

   В консоли будет:
   - список таблиц и колонок для User, Investor, Payment, …, ChatMessage;
   - проверка типа User.id (должен быть integer для текущего Prisma);
   - есть ли внешние ключи у ChatMessage.

4) Результат пришли агенту ТЕКСТОМ из консоли (без паролей в строках
   подключения — можно замазать середину URI). По отчёту можно сказать,
   что безопасно, а что нужно поправить до миграций.

5) После анализа верни .env на «рабочую» базу, если смотрел не ту.

Что НЕ делать без согласования
------------------------------
• Не запускать на старой прод-базе prisma db push (может переписать схему).
• Не менять пароль БД в середине рабочего дня без обновления Vercel.

Команда db:verify — короткая проверка «та ли это база» (есть ли Investor).
Команда db:analyze — подробный снимок схемы под старый проект.
